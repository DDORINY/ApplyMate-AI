# v0.6.0 규칙 기반 채용공고 추천 상세 개발계획

## 1. 버전 목적

v0.6.0은 사용자가 저장한 채용공고와 사용자 커리어 데이터를 비교해 규칙 기반 추천 점수, 등급, 추천 이유, 부족 조건을 제공한다.

## 2. 사용자 가치

사용자는 저장한 공고 중 지금 지원 우선순위가 높은 공고를 빠르게 확인하고, 왜 추천되었는지와 어떤 조건이 부족한지 확인할 수 있다.

## 3. 선행 조건

- 회원/인증
- 커리어 프로필
- 채용공고 저장
- 채용공고 분석
- 사용자-공고 적합도 분석
- 지원 현황

## 4. 구현 범위

- 규칙 기반 추천 생성
- 추천 목록/상세 조회
- 추천 실행 이력
- 추천 이유/부족 조건 저장
- 추천 피드백 생성/수정/삭제
- outdated 판정
- 추천 화면과 내비게이션 연결
- API/DB/문서/테스트

## 5. 제외 범위

- 외부 채용공고 자동 수집
- 머신러닝 추천
- AI 추천 문장 생성
- 실제 지원 자동 수행
- 사용자 승인 없는 지원 상태 변경

## 6. 사용자 흐름

프로필 작성 → 채용공고 저장 → 공고 분석 → 추천 생성 → 추천 목록 확인 → 추천 이유/부족 조건 확인 → 관심 있음/관심 없음/숨기기 피드백 → 지원 준비 또는 공고 상세 이동.

## 7. 추천 입력 데이터

- 커리어 프로필
- 희망 직무
- 희망 고용 형태
- 희망 근무 지역
- 보유 기술
- 경력
- 프로젝트
- 채용공고
- 채용공고 분석
- 기존 적합도 분석
- 지원 이력
- 사용자 피드백

## 8. 추천 제외 조건

- 삭제/보관/마감 공고
- 이미 지원한 동일 공고
- 숨김 또는 관심 없음 피드백이 있는 공고
- 다른 사용자 소유 공고

## 9. 점수 체계

총점은 0~100점이다. Frontend는 점수를 계산하지 않고 Backend 결과만 표시한다.

## 10. 가중치 정책

- 직무 일치: 25점
- 기술 일치: 25점
- 경력 수준 일치: 15점
- 고용 형태 일치: 10점
- 근무 지역 일치: 10점
- 프로젝트/도메인 일치: 10점
- 사용자 선호 및 이력: 5점

정책 버전은 `RULE_BASED_V1`로 저장한다.

## 11. 필수 조건과 선호 조건

필수 조건은 `MATCHED`, `MISSING`, `UNKNOWN`, `NOT_APPLICABLE`로 구분한다. 정보가 없으면 불충족으로 단정하지 않고 `UNKNOWN`으로 처리한다.

## 12. 추천 이유

추천 이유는 계산 결과에서 구조화한다. AI를 호출해 이유를 꾸며내지 않는다.

## 13. 부족 조건

부족 조건은 필수/선호 심각도와 함께 저장한다. 필수 기술 누락은 blocking mismatch로 표시할 수 있다.

## 14. 사용자 피드백

피드백 유형은 `INTERESTED`, `NOT_INTERESTED`, `HIDDEN`, `APPLIED`, `SAVED_FOR_LATER`를 사용한다. 초기 버전은 피드백을 저장만 하고 자동 가중치 학습은 하지 않는다.

## 15. 추천 이력

추천 생성 run을 저장하고, 추천 결과는 run과 연결한다. 동일 입력 hash와 정책 버전의 불필요한 중복 추천은 최신 결과를 재사용하거나 갱신한다.

## 16. DB 설계

- `job_recommendation_runs`
- `job_recommendations`
- `job_recommendation_reasons`
- `job_recommendation_feedback`

모든 테이블은 사용자 소유권 검사를 위한 `user_id` 또는 상위 추천 연결을 가진다.

## 17. API 설계

- `POST /api/v1/recommendations/jobs/generate`
- `GET /api/v1/recommendations/jobs`
- `GET /api/v1/recommendations/jobs/{recommendationId}`
- `POST /api/v1/recommendations/jobs/{recommendationId}/feedback`
- `DELETE /api/v1/recommendations/jobs/{recommendationId}/feedback`
- `POST /api/v1/recommendations/jobs/{recommendationId}/refresh`
- `GET /api/v1/recommendations/jobs/runs`
- `GET /api/v1/recommendations/jobs/runs/{runId}`
- `GET /api/v1/recommendations/jobs/policy`

## 18. Backend 설계

Router, Service, Repository, Policy를 분리한다. 추천 방식은 항상 `RULE_BASED`로 저장하고 응답한다.

## 19. Frontend 설계

- `/recommendations` 추천 목록
- `/recommendations/{recommendationId}` 추천 상세
- 점수/등급/필수 조건/추천 이유/부족 조건/피드백 표시
- 대시보드와 채용공고 상세 연결

## 20. 성능 정책

프로필, 기술, 프로젝트, 공고, 분석, 지원 이력, 피드백은 일괄 조회한다. 공고마다 반복 DB query를 만들지 않는다.

## 21. 사용자 소유권

추천, run, feedback, job 조회와 변경은 모두 `current_user.id` 기준으로 제한한다.

## 22. 오류 처리

공통 오류 구조를 사용하고 내부 예외 메시지를 노출하지 않는다. 일부 공고 실패는 run의 failed count와 safe error로 기록한다.

## 23. 테스트 전략

점수 경계, 기술 정규화, 필수 조건, 지원 완료 공고 제외, 숨김/관심 없음 제외, 피드백 CRUD, outdated, IDOR 차단을 테스트한다.

## 24. Migration 전략

현재 head `20260719_2200` 다음 revision으로 `20260719_2300_create_job_recommendation_tables.py`를 생성한다. upgrade/downgrade 대칭을 검증한다.

- 피드백 기반 자동 학습은 v0.6.0 범위가 아니다.
