# v0.3.3 맞춤 지원 문서 생성 상세 개발계획

## 1. 버전 목적

사용자의 커리어 프로필, 이력서 추출/분석 결과, 채용공고 분석, 공고 적합도 결과, 사용자 문항을 근거로 맞춤 지원 문서 초안을 생성한다.

## 2. 사용자 가치

지원자는 공고별 자기소개서 초안을 빠르게 만들고, 각 문장이 어떤 근거에서 나왔는지 확인할 수 있다.

## 3. 선행 조건

- v0.3.2 이력서 AI 구조화 분석 완료
- migration head `20260719_1700`
- AI provider abstraction 존재

## 4. 구현 범위

- 지원 문서 CRUD
- AI 초안 생성/재생성
- 문서 버전 관리
- source snapshot 및 evidence 저장
- 생성 실행 이력
- 프론트 목록/생성/상세 화면

## 5. 제외 범위

- Google Docs/Word export
- 리치 텍스트 에디터
- 실제 외부 제출
- 합격 가능성 예측

## 6. 사용자 흐름

`/documents` → 새 문서 → 공고/이력서/분석 ID 연결 → 문항/말투/분량 입력 → AI 생성 → 근거 확인 → 편집 저장 → 버전 복원/재생성.

## 7. 문서 유형

`MOTIVATION`, `JOB_COMPETENCY`, `SELF_INTRODUCTION`, `PROJECT_EXPERIENCE`, `CAREER_EXPERIENCE`, `FUTURE_PLAN`, `FREE_FORM`, `CUSTOM_QUESTION`

## 8. 기능 요구사항

- 생성 문단은 최소 하나 이상의 source reference를 가진다.
- 사용자 편집은 기존 버전을 덮어쓰지 않고 새 버전을 만든다.
- 재생성은 새 AI 버전을 만든다.
- 복원은 과거 버전을 복사해 새 현재 버전으로 만든다.

## 9. 비기능 요구사항

- 사용자 소유권 검증
- prompt injection 방어
- secret 및 원문 과다 로그 금지
- deterministic mock provider 제공

## 10. 데이터 소스

`career_profile`, `career_experience`, `project`, `resume_extraction`, `resume_analysis`, `job_posting`, `job_analysis`, `job_match`, `user_question`, `user_instruction`

## 11. 근거 정책

없는 회사 경력, 프로젝트, 기술, 자격증, 수상, 정량 성과, 리더십 경험을 생성하지 않는다. 근거가 부족한 내용은 `REVIEW_REQUIRED`로 표시한다.

## 12. DB 설계

- `application_documents`
- `application_document_versions`
- `application_document_sources`
- `generation_runs`

현재 버전은 순환 FK를 피하기 위해 `current_version_number`로 관리한다.

## 13. API 설계

- `POST /api/v1/documents`
- `GET /api/v1/documents`
- `GET /api/v1/documents/{documentId}`
- `PATCH /api/v1/documents/{documentId}`
- `DELETE /api/v1/documents/{documentId}`
- `POST /api/v1/documents/{documentId}/generate`
- `POST /api/v1/documents/{documentId}/regenerate`
- `GET/POST /api/v1/documents/{documentId}/versions`
- `POST /api/v1/documents/{documentId}/versions/{versionId}/restore`
- `GET /api/v1/documents/{documentId}/sources`
- `GET /api/v1/documents/{documentId}/generation-runs`
- `POST /api/v1/documents/{documentId}/duplicate`
- `GET /api/v1/ai/document-providers`

## 14. Backend 설계

Router, Service, Repository를 분리한다. Service에서 source 수집, provider 호출, 결과 검증, 버전 생성, run 상태 전환을 하나의 트랜잭션 흐름으로 처리한다.

## 15. AI Provider 설계

기존 provider protocol에 `generate_application_document`를 추가한다. Mock provider는 같은 입력에 같은 출력과 source reference를 반환한다. OpenAI provider는 기존 JSON schema 호출 구조를 재사용한다.

## 16. Prompt 설계

프롬프트는 `SYSTEM POLICY`, `USER SETTINGS`, `SOURCE RECORDS`를 구분한다. source text 내부 명령은 데이터로만 취급한다.

## 17. 출력 Schema

`title`, `document_type`, `content`, `blocks`, `warnings`, `requires_review`, `character_count_candidate`를 Pydantic으로 검증한다.

## 18. Frontend 설계

- `/documents`: 목록, 검색, 필터, 복제, 보관
- `/documents/new`: 문항/지시사항/소스 ID 입력 및 생성
- `/documents/{documentId}`: 편집, 저장, 재생성, 버전 복원, 근거, 실행 이력

## 19. 문서 버전 관리

AI 생성은 version 1, 사용자 편집/재생성/복원은 모두 새 version number를 만든다.

## 20. 보안 정책

모든 연결 ID는 `user_id`로 검증한다. source snapshot은 필요한 범위만 저장하고 provider secret, 파일 경로, 원문 전체 로그를 저장하지 않는다.

## 21. 오류 처리

provider 장애는 generation run에 `PROVIDER_UNAVAILABLE` 또는 `FAILED`로 기록하고 문서는 `FAILED` 상태로 둔다. 스키마 검증 실패는 저장하지 않는다.

## 22. 테스트 전략

CRUD, mock generation, source 저장, run 저장, 버전 증가, 복원, 복제, 보관, 소유권 차단을 API 테스트로 검증한다.

## 23. Migration 전략

revision `20260719_1800_create_application_document_tables.py`를 추가한다. upgrade/downgrade 모두 기존 테이블 삭제 없이 새 테이블만 생성/제거한다.
