# v0.4.1 일정 관리 개발 계획

## 1. 버전 목적

v0.4.1은 v0.4.0 지원 현황 관리 위에 일정 관리 기능을 연결한다. 사용자는 지원 마감, 코딩 테스트, 과제, 면접, 결과 발표, 오퍼 응답 기한 등을 하나의 캘린더에서 관리할 수 있다.

## 2. 사용자 가치

사용자는 여러 채용 절차의 마감과 준비 일정을 놓치지 않고, 지원 항목·채용공고와 연결된 일정 맥락을 한 화면에서 확인할 수 있다.

## 3. 선행 조건

- v0.4.0 지원 현황 관리가 완료되어야 한다.
- `applications` 테이블과 `job_postings` 테이블이 존재해야 한다.
- Alembic 기준 head는 `20260719_1900`이다.
- 인증된 사용자 기준의 소유권 검사가 모든 일정 API에 적용되어야 한다.

## 4. 구현 범위

- 일정 생성, 목록, 상세, 수정, 보관 처리
- 일정 상태 변경: 완료, 취소, 예정 상태 변경
- 지원 항목과 채용공고 연결
- 일정 유형, 상태, 신뢰도 관리
- 알림 설정 CRUD
- 월간, 주간, 목록 조회
- 임박 일정과 지난 일정 계산
- 일정 충돌 감지 및 경고
- 일정 변경 이력 저장 및 조회
- 지원 상세와 채용공고 상세 화면의 연결 일정 진입점

## 5. 제외 범위

- 반복 일정
- Google Calendar 실제 일정 생성·수정·삭제
- Gmail/이메일 자동 일정 수집
- 이메일, 푸시, 문자 실제 발송
- 백그라운드 알림 워커
- 일정 생성 시 지원 상태 자동 변경

## 6. 사용자 흐름

1. 사용자는 지원 항목 상세에서 관련 일정을 추가한다.
2. 사용자는 캘린더 화면에서 직접 일정을 생성한다.
3. 사용자는 일정 유형, 시작/종료 시각, 종일 여부, 신뢰도, 알림을 입력한다.
4. 사용자는 월간·주간·목록 보기에서 전체 일정을 확인한다.
5. 사용자는 일정 상세에서 수정, 완료, 취소, 보관 처리를 수행한다.
6. 사용자는 일정 상세에서 변경 이력과 알림 목록을 확인한다.

## 7. 일정 이벤트 유형

```text
APPLICATION_DEADLINE
DOCUMENT_RESULT
CODING_TEST
ASSIGNMENT_DEADLINE
INTERVIEW
FINAL_INTERVIEW
FINAL_RESULT
OFFER_RESPONSE_DEADLINE
USER_EVENT
```

v0.4.1에서는 위 유형만 구현한다. 문서 제출, 포트폴리오 제출, 리크루터 콜, 오리엔테이션은 실제 사용자 흐름이 필요해지는 버전에서 추가한다.

## 8. 일정 상태

```text
SCHEDULED
CONFIRMED
COMPLETED
CANCELLED
MISSED
TENTATIVE
```

- 생성 기본값은 `SCHEDULED`이다.
- 사용자가 확정한 일정은 `CONFIRMED`로 변경할 수 있다.
- 완료된 일정은 `COMPLETED`로 변경한다.
- 취소된 일정은 `CANCELLED`로 변경한다.
- `MISSED`는 별도 백그라운드 작업 없이 조회 시점에 계산 필드로 표시한다.
- 상태와 일정 신뢰도는 서로 다른 개념으로 분리한다.

## 9. 일정 신뢰도

```text
CONFIRMED
ESTIMATED
USER_INPUT
AI_EXTRACTED
EMAIL_EXTRACTED
```

- 사용자가 직접 입력한 일정은 기본 `USER_INPUT`이다.
- 공고나 공식 안내에 명시된 일정은 `CONFIRMED`로 표시한다.
- 시스템 또는 AI가 추정한 일정은 `ESTIMATED` 또는 `AI_EXTRACTED`로 표시한다.
- 이메일 자동 추출은 v0.4.1 제외 범위이지만 향후 호환을 위해 `EMAIL_EXTRACTED` 값을 예약한다.

## 10. 기능 요구사항

- 일정은 제목, 유형, 시작 시각, 종료 시각, 종일 여부를 필수로 가진다.
- `start_at < end_at`을 만족해야 한다.
- 일정은 `application_id`, `job_id` 중 하나 이상과 연결될 수 있고, 둘 다 없어도 사용자 개인 일정으로 저장할 수 있다.
- `application_id`와 `job_id`가 함께 제공되면 지원 항목의 `job_id`와 일치해야 한다.
- 보관된 일정은 기본 목록에서 제외되며 `include_archived=true`일 때만 포함한다.
- 취소·완료된 일정의 알림은 비활성으로 간주한다.
- 충돌은 저장을 차단하지 않고 경고 목록으로 제공한다.

## 11. 비기능 요구사항

- 모든 사용자 데이터 조회에는 `user_id` 소유권 검사를 적용한다.
- API 응답은 기존 공통 응답 구조를 유지한다.
- 목록 API는 페이지네이션과 필터를 지원한다.
- 시간 정보는 ISO 8601 형식으로 반환한다.
- 실제 발송되지 않는 알림을 발송 완료처럼 표시하지 않는다.
- 프론트엔드는 로딩, 빈 상태, 오류 상태를 제공한다.

## 16. 타임존 정책

- DB에는 timezone-aware UTC datetime을 저장한다.
- 프론트엔드는 사용자 타임존 기준으로 표시한다.
- 기본 타임존은 `Asia/Seoul`이다.
- API는 ISO 8601 문자열을 사용한다.
- 종일 마감 일정은 `all_day=true`, `start_at`은 해당 날짜 00:00, `end_at`은 다음 날짜 00:00으로 저장한다.

## 17. 알림 정책

- 지원 프리셋: 10분, 30분, 1시간, 3시간, 1일, 3일, 1주, 직접 입력
- `minutes_before`는 0보다 커야 한다.
- `scheduled_at`은 일정 시작 전이어야 한다.
- 같은 일정에 같은 `reminder_type`과 `minutes_before` 조합을 중복 저장하지 않는다.
- v0.4.1은 `IN_APP` 저장과 표시만 구현한다.
- `EMAIL`, `PUSH`는 enum 호환만 유지하고 실제 발송은 후속 버전에서 구현한다.

## 18. 충돌 정책

충돌은 같은 사용자의 취소·보관되지 않은 일정 중 시간 범위가 겹치는 경우로 정의한다.

```text
existing.start_at < new.end_at AND existing.end_at > new.start_at
```

경계만 맞닿는 일정은 충돌이 아니다. 충돌은 저장을 차단하지 않고 경고로 제공한다.

## 19. 소유권 정책

- 일정, 알림, 변경 이력은 모두 `user_id`를 가진다.
- 다른 사용자의 일정·알림·이력은 조회, 수정, 삭제할 수 없다.
- 연결된 지원 항목과 채용공고도 현재 사용자 소유 데이터만 허용한다.

## 26. 알려진 리스크

- 실제 알림 발송 워커가 없으므로 v0.4.1 알림은 저장·표시 기능에 한정된다.
- 반복 일정이 없어 장기 채용 프로그램 일정은 수동으로 여러 개 생성해야 한다.
- 타임존과 종일 일정은 프론트 표시 정책과 API 저장 정책이 어긋나지 않도록 회귀 테스트가 필요하다.
- Google Calendar 연동 전까지 외부 캘린더와의 동기화 상태는 제공하지 않는다.
