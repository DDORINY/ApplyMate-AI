# v0.9.0 E2E·성능·보안 안정화 상세 계획

## 1. 목적

v1.0.0 MVP 릴리스 전, 신규 대형 기능보다 현재 구현된 전체 흐름의 안정성·관측성·운영 준비도를 높인다.

## 2. 현재 시스템 범위

- 기준 완료 버전: `v0.8.0`
- 작업 브랜치: `feature/v0.9.0-stability-hardening`
- 완료 후 버전: `v0.9.0`
- 완료 후 migration head: `20260720_0300`
- 주요 대상: 인증, 프로필, 공고, 분석, 이력서, 지원 문서, 지원 현황, 일정, 연동, 추천, 알림, worker

## 3. 구현 범위

- Request ID middleware
- 공통 오류 응답 `request_id`
- 보안 헤더
- CORS origin 환경변수
- 주요 API rate limit
- `/health/live`, `/health/ready`
- 운영 필수 설정 readiness 검증
- 범용 감사 로그 `audit_logs`
- Playwright E2E smoke
- E2E/성능/배포/백업 문서

## 4. 제외 범위

- 실제 운영 부하 테스트 완료 선언
- 운영 DB 복구 리허설 완료 선언
- 실제 SMTP/Push 실발송 완료 선언
- 실제 OpenAI/Google Calendar/Gmail 운영 호출 완료 선언
- 대규모 인증 토큰 저장 구조 변경

## 5. 브라우저 E2E

Playwright 기반 smoke test를 추가한다.

- 로그인 화면 접근
- 회원가입 화면 접근
- 인증 없는 보호 화면 안내
- 알림 설정 화면 접근

도메인별 전체 데이터 생성 E2E는 v1.0.0 인수 테스트에서 확장한다.

## 6. API 통합 테스트

기존 backend 전체 테스트를 유지하고 v0.9.0 운영 안정화 테스트를 추가한다.

- Request ID/error response
- security headers
- CORS preflight
- rate limit 429/header
- notification settings audit log

## 7. 성능 테스트

성능 기준과 대상 endpoint를 문서화한다.

- `GET /api/v1/dashboard`
- `GET /api/v1/jobs`
- `GET /api/v1/applications`
- `GET /api/v1/calendar/events`
- `GET /api/v1/recommendations/jobs`
- `GET /api/v1/notifications`

실제 수치를 측정하지 않은 항목은 완료로 표시하지 않는다.

## 8. DB 성능

v0.9.0에서는 신규 감사 로그 인덱스를 추가한다.

- `(user_id, action, created_at)`
- `(resource_type, resource_id)`
- `request_id`

기존 대량 목록 API 인덱스는 문서 점검 대상으로 남긴다.

## 9. Rate Limit

인메모리 기반 middleware로 주요 민감 API를 제한한다. Redis 기반 분산 rate limit은 운영 확장 항목이다.

- signup/login/refresh
- 비밀번호 재설정·이메일 재발송 후보
- 추천 generate/run-if-due
- notification run-due

## 10. 보안 헤더

기본 적용:

- `Content-Security-Policy`
- `X-Content-Type-Options`
- `Referrer-Policy`
- `Permissions-Policy`
- `X-Frame-Options`

HSTS는 HTTPS 운영 환경에서 `HSTS_ENABLED=true`로 적용한다.

## 11. 인증·세션 점검

기존 테스트가 access token, refresh rotation, reuse reject, logout, 비밀번호 변경 후 세션 처리를 검증한다. v0.9.0에서는 rate limit과 request id로 운영 추적성을 보강한다.

## 12. 파일 업로드 보안

기존 이력서 업로드 테스트가 확장자, MIME, signature, DOCX 구조, 다중 확장자, path filename, 크기 제한, 중복 hash를 검증한다. v0.9.0에서는 정책 문서와 배포 체크리스트에 반영한다.

## 13. OAuth 보안

기존 Google/GitHub 로그인 OAuth, Calendar OAuth, Gmail OAuth는 state 만료·1회 사용·redirect 제한·token 비노출 테스트를 유지한다. 실제 provider 검증은 `NEEDS_VERIFICATION`이다.

## 14. Worker 안정화

v0.8.0 one-shot worker와 notification processing run을 유지한다. v0.9.0에서는 중복 notification을 DB deduplication key와 테스트로 확인하고, 장기 scheduler/분산 lock은 v1.0.0 운영 검증 항목으로 남긴다.

## 15. 로그·감사

`audit_logs`를 추가한다.

- `user_id`
- `action`
- `resource_type`
- `resource_id`
- `result`
- `request_id`
- `ip_hash`
- `user_agent`
- `audit_metadata`

v0.9.0 연결 액션:

- `NOTIFICATION_SETTINGS_UPDATED`
- `NOTIFICATION_DELIVERY_RETRIED`

## 16. 오류·관측성

모든 공통 오류 응답에 안전한 `request_id`를 포함한다. stack trace, token, secret, raw provider error는 사용자 응답에 노출하지 않는다.

## 17. 운영 설정 검증

`/health/ready`에서 운영 환경 기준 필수 설정을 점검한다.

- DB/Redis
- JWT secret
- URL 설정
- CORS wildcard 금지
- provider별 credential

mock/disabled provider는 live credential을 강제하지 않는다.

## 18. Docker 안정화

Compose backend healthcheck를 `/api/v1/health/ready`로 변경한다. 신규 운영 환경변수를 backend service에 전달한다. 기존 volume은 삭제하지 않는다.
