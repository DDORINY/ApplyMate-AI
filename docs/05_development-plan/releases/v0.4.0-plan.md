# v0.4.0 지원 현황 관리 상세 개발 계획

## 1. 버전 목적

채용공고, 이력서, 지원 문서 버전을 실제 지원 항목으로 연결하고 사용자가 지원 상태를 추적할 수 있게 한다.

## 2. 사용자 가치

사용자는 “저장한 공고”를 넘어 준비 중, 지원 완료, 서류 검토, 면접, 오퍼, 불합격, 종료까지 전 과정을 한 화면에서 관리할 수 있다.

## 3. 선행 조건

- v0.3.3 지원 문서 생성 완료
- `job_postings`, `resumes`, `resume_files`, `application_documents`, `application_document_versions` 존재
- Alembic head: `20260719_1800`

## 4. 구현 범위

- 지원 항목 CRUD 및 보관
- 상태 변경 및 상태 변경 이력 저장
- 지원 메모 CRUD
- 공고/이력서/이력서 파일/지원 문서/문서 버전 연결
- 검색, 필터, 정렬, 페이지네이션
- `/applications`, `/applications/new`, `/applications/{applicationId}` 화면
- 지원 문서 생성 화면의 직접 숫자 ID 입력 제거 및 선택 UI 적용

## 5. 제외 범위

- 외부 채용 사이트 자동 지원
- Google Calendar 실제 일정 생성
- 이메일 자동 수집에 의한 상태 변경
- 합격 가능성 예측

## 6. 사용자 흐름

1. 사용자가 채용공고 상세에서 “지원 관리에 추가”를 누른다.
2. 지원 항목 생성 화면에서 공고, 이력서, 제출 문서 버전을 선택한다.
3. 상태, 지원 예정일/완료일, 지원 경로, 담당자 정보를 입력한다.
4. 상세 화면에서 상태 변경, 메모 작성, 관련 공고/문서 이동을 수행한다.

## 7. 지원 상태

```text
SAVED
PREPARING
APPLIED
DOCUMENT_REVIEW
CODING_TEST
ASSIGNMENT
INTERVIEW
FINAL_INTERVIEW
OFFER
REJECTED
WITHDRAWN
CLOSED
```

## 8. 상태 전환 정책

권장 흐름은 `SAVED → PREPARING → APPLIED → DOCUMENT_REVIEW → INTERVIEW → OFFER/CLOSED`이지만, 실제 채용 절차는 회사마다 다르므로 v0.4.0은 모든 유효 enum 상태로 이동을 허용한다. 모든 변경은 `application_status_history`에 기록한다.

## 9. 기능 요구사항

- 지원 항목 생성/조회/수정/보관
- 상태 변경 시 이전 상태, 새 상태, 사유, 메모, 변경 출처 저장
- 메모 생성/수정/삭제/고정 여부 저장
- 지원 문서 버전은 `application_document_version_id`로 고정
- 현재 사용자 소유 데이터만 연결 가능

## 10. 비기능 요구사항

- 사용자 소유권 검증으로 IDOR 방지
- 목록 API 페이지네이션 제공
- 주요 필터 컬럼 인덱스 생성
- 민감정보와 토큰을 로그에 남기지 않음

## 13. Backend 설계

- Model: `app.models.application`

- Repository: `ApplicationRepository`
- Service: `ApplicationService`
- Router: `/api/v1/applications`

## 14. Frontend 설계

- 목록: `/applications`
- 생성: `/applications/new`
- 상세: `/applications/{applicationId}`
- 공통 선택 컴포넌트: 공고, 이력서, 이력서 파일, 분석, 지원 문서, 문서 버전 선택

## 15. 소유권 정책

모든 연결 리소스는 `user_id`로 현재 사용자 소유 여부를 확인한다. 다른 사용자 리소스는 `404`로 처리한다.

## 16. 오류 정책

- `APPLICATION_NOT_FOUND`
- `APPLICATION_NOTE_NOT_FOUND`
- `APPLICATION_INVALID_RELATION`
- 기존 `JOB_NOT_FOUND`, `RESUME_NOT_FOUND`, `DOCUMENT_NOT_FOUND`, `DOCUMENT_VERSION_NOT_FOUND` 재사용

## 17. 검색/필터/정렬

- keyword, status, company, job_id, priority, application_channel
- applied_from/applied_to, updated_from/updated_to
- has_document, has_resume, archived
- sort: created_at, updated_at, applied_at, planned_apply_at, company_name, status, priority

## 18. 테스트 전략

- CRUD/list/detail/update/archive
- 상태 변경 이력
- 메모 생성/수정/삭제
- 연결 관계 불일치 검증
- 사용자 소유권 검증
- 문서 버전 고정 검증
- options 사용자 데이터 한정 검증

## 19. 마이그레이션 전략

새 테이블만 추가한다. 기존 컬럼 삭제나 변경은 없다. PostgreSQL enum은 `checkfirst=True`로 생성하고 downgrade 시 테이블 삭제 후 enum을 제거한다.
