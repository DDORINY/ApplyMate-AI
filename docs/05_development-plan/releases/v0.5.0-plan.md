# v0.5.0 Google Calendar 연동 상세 개발계획

## 1. 버전 목적

ApplyMate AI 내부 일정과 Google Calendar를 연결할 수 있는 기반을 구현한다. 로그인용 Google OAuth와 Calendar 연동용 OAuth를 분리하고, 사용자의 명시적 승인 없이 외부 Calendar에 일정을 생성/수정/삭제하지 않는다.

## 2. 사용자 가치

사용자는 ApplyMate AI에 등록한 지원 마감, 면접, 코딩테스트, 과제, 결과 발표 일정을 Google Calendar에서도 확인할 수 있다.

## 3. 선행 조건

- v0.4.1 일정 관리 완료
- v0.4.2 대시보드 완료

- 실제 Google credentials가 없어도 `CALENDAR_PROVIDER=mock`으로 검증 가능해야 한다.

## 6. 사용자 흐름

1. 사용자가 로그인한다.
2. `/settings/integrations`로 이동한다.
3. Google Calendar 연결을 시작한다.
4. OAuth 동의 후 callback으로 연결을 완료한다.
5. Calendar 목록에서 writable Calendar를 선택한다.
6. 동기화 방향과 활성화 여부를 저장한다.
7. 내부 일정 상세 또는 전체 동기화를 실행한다.
8. sync mapping, run, error 상태를 확인한다.
9. 필요하면 Calendar 연결을 해제한다.

## 7. 로그인 OAuth와 Calendar OAuth 분리

- 로그인 OAuth는 사용자 인증과 기본 프로필 식별만 담당한다.
- Calendar OAuth는 Calendar 목록 조회와 일정 생성/수정/삭제 권한만 담당한다.
- 기존 `oauth_accounts`는 Calendar token 저장에 사용하지 않는다.
- Calendar 연결은 별도 `external_accounts`와 `calendar_connections`에 저장한다.

## 8. OAuth Scope

초기 scope:

- `openid`
- `email`
- `profile`
- `https://www.googleapis.com/auth/calendar.calendarlist.readonly`
- `https://www.googleapis.com/auth/calendar.events`

실제 Google OAuth consent와 scope 적정성은 운영 credentials가 준비된 뒤 `NEEDS_VERIFICATION`으로 검증한다.

## 9. Token 저장 정책

- access token, refresh token, sync token은 평문 저장하지 않는다.
- `EXTERNAL_TOKEN_ENCRYPTION_KEY`로 application-level encryption을 수행한다.
- token key version은 `EXTERNAL_TOKEN_ENCRYPTION_KEY_VERSION`으로 기록한다.
- token 원문은 API 응답과 로그에 노출하지 않는다.
- 연결 해제 시 token은 폐기/비활성화한다.

## 11. Calendar 선택 정책

- 연결 완료 후 Calendar 목록을 조회한다.
- writable Calendar만 동기화 대상으로 선택할 수 있다.
- read-only Calendar 선택 시 `CALENDAR_NOT_WRITABLE` 오류를 반환한다.
- 기본 mock provider는 primary writable Calendar를 자동 선택한다.

## 12. 동기화 방향

지원 값:

- `INTERNAL_TO_GOOGLE`
- `GOOGLE_TO_INTERNAL`
- `BIDIRECTIONAL`

v0.5.0의 검증 완료 범위는 `INTERNAL_TO_GOOGLE` mock sync다. Google → 내부 자동 import는 충돌/중복 위험 때문에 후속 버전에서 사용자 확인 기반으로 확장한다.

## 13. 동기화 Mapping

`calendar_sync_mappings`가 내부 일정과 외부 event를 연결한다.

중복 방지:

- `connection_id + external_event_id`
- `connection_id + schedule_event_id`

기존 mapping이 있으면 새 외부 event를 만들지 않고 update 흐름을 사용한다.

## 14. 충돌 정책

양쪽이 마지막 동기화 이후 모두 변경된 경우 `CONFLICT` 상태로 둔다. 자동 덮어쓰기는 하지 않는다.

초기 UI는 충돌 요약과 수동 재동기화 안내를 제공하고, 필드별 충돌 해결은 후속 버전에서 확장한다.

## 15. 삭제 정책

- 내부 일정 삭제/보관이 곧바로 Google Calendar hard delete로 이어지지 않는다.
- 외부 Calendar 삭제가 내부 일정 hard delete로 이어지지 않는다.
- 연결 해제 시 내부 일정과 mapping은 보존하고 connection/token만 비활성화한다.

## 20. 보안 정책

- 로그인 OAuth token과 Calendar OAuth token을 섞지 않는다.
- Calendar token은 암호화 저장한다.
- secret과 token 원문은 commit하지 않는다.
- raw provider response를 저장하지 않는다.
- 모든 연결, mapping, sync run/error는 `user_id`로 소유권을 제한한다.
- OAuth state는 hash 저장, 만료, 1회 사용을 적용한다.

## 27. 알려진 위험

- 실제 Google OAuth/Calendar API는 운영 credentials가 필요하다.
- Google API quota와 consent screen 설정에 따라 운영 동작이 달라질 수 있다.
- stdlib 기반 token encryption은 의존성 추가를 피한 v0.5.0 구현이며, 운영 전 보안 검토가 필요하다.
- Google → 내부 자동 import는 중복/오탐 위험이 있어 후속 버전에서 사용자 확인 기반으로 확장한다.
