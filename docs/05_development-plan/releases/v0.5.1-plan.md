# v0.5.1 Gmail 채용 메일 분석 상세 개발계획

## 1. 버전 목적

사용자가 명시적으로 연결한 Gmail 계정에서 채용 관련 메일을 읽기 전용으로 조회하고, 지원 상태와 일정에 반영할 수 있는 후보를 생성한다.

## 2. 사용자 가치

- 지원 접수, 결과, 면접, 과제, 코딩테스트 메일을 한곳에서 검토한다.
- 메일 근거를 보고 지원 현황 또는 일정에 반영할지 결정한다.
- 자동 반영을 막아 개인정보와 데이터 정확성을 지킨다.

## 3. 선행 조건

- v0.5.0 Google Calendar 연동 기반 완료
- `external_accounts`와 외부 토큰 암호화 구조 사용
- 사용자 인증, 지원 현황, 일정 관리 기능 완료

## 4. 구현 범위

- Gmail OAuth 연결
- Gmail 읽기 전용 scope
- Gmail provider interface
- Mock Gmail Provider
- Google Gmail Provider 골격
- 메일 조회와 중복 저장 방지
- 규칙 기반 mock 채용 메일 분석
- 이메일 후보 생성
- 지원 항목 연결 후보
- 사용자 승인 기반 상태 변경
- 사용자 승인 기반 일정 생성
- 프론트 Gmail 연동 화면
- 프론트 메일 후보 목록 화면
- 테스트와 문서 최신화

## 5. 제외 범위

- Gmail 메일 수정, 삭제, 발송
- 사용자 승인 없는 지원 상태 변경
- 사용자 승인 없는 일정 생성
- Gmail 전체함 대규모 수집
- 첨부파일 다운로드/저장
- 실제 Gmail API 운영 검증 완료 선언

## 6. 사용자 흐름

1. `/settings/integrations`에서 Gmail 연결을 시작한다.
2. OAuth 동의를 완료한다.
3. Gmail 동기화를 실행한다.
4. 채용 관련 메일 후보를 검토한다.
5. 필요하면 지원 항목을 연결한다.
6. 상태 변경 또는 일정 생성을 명시적으로 승인한다.
7. 승인된 항목만 내부 데이터에 반영된다.

## 7. OAuth 분리

OAuth 목적은 `LOGIN`, `CALENDAR`, `GMAIL`로 분리한다. Google 로그인 또는 Calendar 연결이 있어도 Gmail은 자동 연결하지 않는다.

## 8. Gmail Scope

사용 scope:

```text
openid
email
profile
https://www.googleapis.com/auth/gmail.readonly
```

금지 scope:

```text
gmail.modify
gmail.send
mail.google.com
```

## 9. 메일 조회 정책

- 기본 조회 기간은 최근 30일이다.
- 기본 최대 조회 건수는 50건이다.
- 검색 query는 채용 관련 키워드 중심이다.
- 전체 Inbox 무제한 수집은 하지 않는다.
- 동일 메일은 `connection_id + provider_message_id` unique 제약으로 중복 저장하지 않는다.

- `OTHER`

## 11. 분석 정책

메일 본문은 신뢰할 수 없는 입력으로 처리한다. prompt injection 문구는 분석 지시가 아니라 데이터로 취급한다.

## 12. 연결 후보 정책

회사명, 직무명, 이메일 발신자, 지원 시점 등을 근거로 지원 항목 후보를 만든다. 자동 연결은 금지한다.

## 13. 사용자 승인 정책

다음 작업은 사용자의 명시적 승인 후에만 수행한다.

- 지원 상태 변경
- 일정 생성
- 지원 항목 연결
- Google Calendar 동기화

## 14. Token 저장 정책

Gmail access token, refresh token, history id는 평문 저장하지 않는다. `EXTERNAL_TOKEN_ENCRYPTION_KEY`가 없으면 provider를 활성화하지 않는다.

## 15. Provider 설계

- `DisabledGmailProvider`
- `MockGmailProvider`
- `GoogleGmailProvider`

v0.5.1 검증은 Mock Provider 기준으로 완료한다. 실제 Gmail API는 `NEEDS_VERIFICATION`이다.

## 18. Backend 설계

Router, Service, Repository, Provider 책임을 분리한다. 승인 API는 하나의 transaction에서 후보 상태, 지원 상태 이력, 일정 생성, action 기록을 처리한다.

## 19. Frontend 설계

- `/settings/integrations`: Gmail 연결, 설정, 동기화, 최근 실행 결과
- `/inbox-candidates`: 후보 목록, 근거, 확인/거절

## 20. 개인정보 보호

- 메일 원문 전체 HTML, inline image, 첨부파일은 저장하지 않는다.
- 저장 대상은 metadata, snippet, 제한된 sanitized text, hash, 후보 근거이다.
- API 응답에 token을 포함하지 않는다.

## 21. 오류 처리

Gmail provider raw error와 메일 원문은 사용자 응답에 노출하지 않는다. 안전한 error code와 message만 반환한다.
